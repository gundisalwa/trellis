The quiet failure mode

An agent that grades its own work will always pass.

Autonomy scales output. Left ungoverned, it also scales the ways a process erodes — silently, one shortcut at a time.

// drift 01

Gates get skipped

The review step is optional, so under deadline it quietly doesn't happen — and nothing records that it didn't.

// drift 02

Ambiguity gets guessed

A vague spec isn't clarified; it's resolved by whichever interpretation the model reached for first.

// drift 03

The builder judges itself

The same agent that wrote the code decides whether the code is good. There is no independent check.

// drift 04

Provenance evaporates

Six agents and forty commits later, no one can say which decision a line of code actually came from.

What it is

Not a methodology. The layer above the one you already use.

Trellis doesn't replace Spec Kit, BMAD, or your team's homegrown workflow. It sits on top, learns the shape of your process, and enforces a small set of invariants — the handful of properties that must hold no matter how fast the agents move.

It augments, never clobbers. Your instructions stay yours; Trellis composes onto them and records any change it proposes.

“A structure that enables growth rather than dictating form.”

The maximally-permissive supervisor: constrain the space, permit maximal freedom within it. Agents move fast — they just can't move off the rails.

How it works

Four parts. A small gate, a working layer, two dials, two floors.

The model is deliberately tiny — small enough that a newcomer, human or agent, can read it and know how to make a change that will pass.

A · Structural gate

Can this be supervised?

A four-point admission check: does the process have one-way flow, handover points, a human intent locus, and checkable artifacts? If not, Trellis says so — loudly.

B · Operating layer

What Trellis supplies

A gate at every handover, independent verification, an auditable archive, bounded context, clarify-before-commit. The guarantees you get because you adopted it.

C · Two dials

How strict, and who checks

Per gate: enforcement strength (documented → default-on → enforced) and gatekeeper (an agent, a human, or nobody). The same core serves a two-session hack and a regulated pipeline.

D · Two floors

The lines that never move

Two settings can never dial to zero: every consequential choice is surfaced, and the human intent gate never fully opens. Everything else bends; these don't.

Think of it as gene expression.

The invariants are a shared genome. Each project switches on a different subset, at a different strength, gatekept by whoever it trusts — a per-project expression profile. A lightweight repo expresses the behavioral genes and leaves the pipeline machinery silent; a regulated org expresses all of them, enforced. One set of rules, many living shapes.

Two ways to run it

Consult it, or install it.

The same invariants, delivered along a dial — from a repo your agents simply read, to a live supervisor wired into your pipeline.

Advisor · pull

Consult it — no install.

Trellis is never installed. Your agents consult it as a reference and internalize what applies. Today: point them at the open repo. Where it's headed: a defined CLI your host invokes, so the interface is crisp rather than "read whatever you find."

  • Zero install, zero runtime
  • Lands as reviewable pull requests
  • A defined CLI interface (in progress)
Supervisor · push

Wire it into the pipeline.

Trellis is installed and live — gates fire on your commit and PR events, it stays current through the update channel, and it comes off cleanly when you want it to.

  • Live gates via hooks, no runtime service
  • Assess a repo, then apply the minimal overlay
  • Self-regulating across your instances

Questions

The reasonable objections.

Does it replace my methodology?

No. Trellis is the layer above your process. Keep Spec Kit, BMAD, or your own workflow — Trellis supervises whichever you use and enforces a small set of invariants on top. It augments your instructions; it never overwrites them.

What does it actually run on?

Instructions, not a runtime. There's no service to operate. Advisor mode is a repo your agents read; Supervisor mode is a Claude Code plugin whose checks fire via hooks, plus an optional CLI. No Python or Node process sits in your critical path.

Is my code sent anywhere?

In Advisor mode, nothing is installed and nothing leaves your environment — your agents just consult the reference. Supervisor and Enterprise tiers are explicit about what they touch, and Enterprise can run fully self-hosted or air-gapped.

Won't this lock me in?

The core is Apache-2.0 — fork it if we disappear. Trellis composes onto your files rather than rewriting them, and uninstalling is clean. Lock-in would violate the very transparency floor the product is built on.

Why should I trust a set of invariants written by one person?

You shouldn't, blindly — and we say so on the page above. The set is labeled "our synthesis," gate-tested against real frameworks, and every claim traces to a concrete artifact. Its generalization is exactly what instance #2 is meant to test.

Can I use it today?

Yes, in Advisor mode: clone the open repo and point your agents at it. The installed Supervisor — and the paid tiers — are in early access while the delivery machinery is built in the open.
